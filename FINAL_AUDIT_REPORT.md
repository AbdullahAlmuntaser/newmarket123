## Technical Audit & ERP Gap Analysis Report

تم إجراء فحص شامل للنظام (Flutter + Drift) وتحليل منطق الأعمال المحاسبي والمخزني. هذا التقرير يوضح الحالة الراهنة، الثغرات، والتوصيات الفنية.

# 1️⃣ تحليل البنية المعمارية (Architecture)

**التقييم: 8.5/10**

- **الهيكل العام:** يتبع المشروع نمط **Clean Architecture** معدل (Data, Domain, Presentation) مع استخدام قوي لـ **Dependency Injection (GetIt)** و **State Management (Bloc/Provider)**.
- **التنظيم:** تقسيم المميزات (Features) ممتاز ومنظم بشكل يسهل التوسع (Scalability).
- **التواصل:** استخدام **EventBusService** لفك الارتباط (Decoupling) بين العمليات (مثل ترحيل المبيعات وتأثيرها على المحاسبة) هو قرار معماري محترف.
- **الاستمرارية:** الكود مكتوب بجودة عالية مع مراعاة Null Safety واستخدام مكتبات حديثة (Decimal, Drift, GoRouter).

# 3️⃣ تحليل الربط وتدفق البيانات (Integration Flow)

تتبعنا عملية البيع من الواجهة إلى القاعدة:
1. **UI:** `PosPage` يرسل حدث `CheckoutEvent`.
2. **Bloc:** `PosBloc` يجهز البيانات ويستدعي `SalesDao`.
3. **Engine:** `TransactionEngine.postSale` يعالج الدفعات (FEFO) ويحدث المخزون.
4. **Event:** يتم إطلاق `SaleCreatedEvent`.
5. **Accounting:** `AccountingService` يستلم الحدث ويرحل القيود آلياً عبر `PostingEngine`.

**الفجوة المكتشفة:**
- **Stock Mismatch:** هناك تحذير (Warning) في الكود يشير إلى احتمال عدم تطابق المخزون مع الدفعات عند المرتجعات.
- **Atomic Updates:** تحديث المخزون يتم برمجياً وليس عبر SQL Atomic Increment، مما قد يسبب Race Conditions في بيئة متعددة المستخدمين (رغم أن المشروع حالياً SQLite محلي).

# 4️⃣ تحليل قاعدة البيانات (Drift/SQLite)

**التقييم: 9/10**

- **التصميم:** قاعدة بيانات ضخمة ومتكاملة (أكثر من 50 جدولاً).
- **العلاقات:** استخدام ممتاز لـ Foreign Keys الموثقة برمجياً.
- **المزامنة:** وجود جدول `SyncQueue` و `SyncableTable` يشير إلى جاهزية النظام للربط السحابي (Cloud Sync).
- **النزاهة:** العمليات المالية مغلفة داخل `db.transaction`.

# 5️⃣ تحليل النظام المحاسبي و ERP Logic

### ✅ الإيجابيات:
1. **Posting Engine:** نظام احترافي يعتمد على "ملفات الترحيل" (Posting Profiles) لتحديد الحسابات المتأثرة.
2. **Costing:** دعم تكاليف الهبوط (Landed Costs) وتخصيصها على الأصناف.
3. **Valuation:** دعم FEFO (الأقدم صلاحية أولاً) وهو مثالي للسوبر ماركت.
4. **ZATCA:** تطبيق Phase 1 (QR Code TLV) بشكل صحيح.

### ❌ النواقص الحرجة (Gaps):
1. **Bill-wise Accounting:** لا يوجد ربط بين السندات والفواتير. العميل يدفع مبلغاً إجمالياً يخصم من حسابه دون تحديد أي فاتورة تم سدادها.
2. **Financial Closing:** نظام إغلاق الفترات موجود برمجياً كجدول ولكن الواجهة والمنطق الفعلي لنقل الأرصدة للسنة الجديدة ناقص.
3. **Advanced Tax:** دعم الضريبة موجود، لكن تقرير الإقرار الضريبي الرسمي (VAT Return) يحتاج لمزيد من التفصيل.

# 6️⃣ الأمن والثغرات (Security & Risks)

1. **Database Encryption:** قاعدة البيانات غير مشفرة (Plain SQLite). أي وصول للملف يعني كشف كل البيانات المالية.
2. **Permissions:** نظام الصلاحيات موجود (RBAC) لكنه يحتاج لربط أدق مع الواجهات (Disable buttons based on role).
3. **Audit Log:** نظام المراجعة يسجل العمليات، لكنه لا يسجل "القيم القديمة والجديدة" (Old/New values)، فقط يسجل الفعل.

---

# 7️⃣ قائمة المشاكل الحرجة (Critical Issues)

1. **Race Condition in Stock:** تحديث المخزون في `postSale` يعتمد على قراءة القيمة ثم طرحها برمجياً ثم كتابتها.
2. **Cloud Backup Incomplete:** ميزة Google Drive Backup هي مجرد هيكل (Stub) ولا تقوم برفع الملف فعلياً.
3. **Sale Return Logic:** منطق إعادة المخزون للدفعات (Batches) يحتاج مراجعة دقيقة لضمان عدم تكرار الدفعات أو ضياع التكلفة.

---

# 8️⃣ خطة التحسين (Refactor Roadmap)

1. **المرحلة 1 (استقرار):**
   - تحويل تحديثات المخزون إلى Atomic SQL (e.g., `stock = stock - ?`).
   - معالجة ثغرة "Stock Mismatch" في المرتجعات.
2. **المرحلة 2 (محاسبة متقدمة):**
   - إضافة "Allocation" للسندات لربطها بالفواتير.
   - تفعيل تقرير ضريبة القيمة المضافة الرسمي.
3. **المرحلة 3 (أمن واتصال):**
   - تشفير قاعدة البيانات باستخدام `sqlcipher`.
   - استكمال ربط Google Drive للنسخ الاحتياطي.
   - تفعيل الـ Cloud Sync الفعلي.
